- [x] Fix bathroom fixtures surviving into non-bathroom redesigns — root cause found on round 4. Rounds 1-3 (prompt wording x2, then raising `guidance_scale`) all failed; user's round-3 test still showed a shower head, now fused onto the bed frame. Round 4: looked up fal.ai's actual flux-pro/kontext API docs and found `strength` (0-1, **default 0.1**) — a parameter the app never set at all, on any of its three generation flows. `strength` caps how much the output is *allowed* to differ from the source image, completely separate from `guidance_scale` (which only controls prompt adherence); at the default 0.1 the model was barely permitted to change the photo no matter how the prompt was worded, which is why every wording/guidance_scale fix fell short. Added `strength` support to `api/generate.js` and set it per-flow on the client: `0.9` for full room strip-and-rebuild (`lamsa-bilingual.html`, `rearrange.html` Full Redesign) and `0.6` for `rearrange.html` Rearrange Only (needs to visibly move furniture while the prompt asks it to keep the furniture's appearance). **User-confirmed**: stress-tested with a genuine bathroom photo converted to a Kids Room (about as hard a case as this gets) — succeeded ~90%. **Round 5**: user re-ran the exact same photo and the floor-to-ceiling shower frame was still there, still fused onto the bed. Since there's no fixed seed, output varies run-to-run at the same settings, but `0.9` clearly wasn't reliably enough for this specific extreme source image (a full glass/metal shower enclosure spanning floor-to-ceiling — plausibly reads to the model as a "structural" element worth preserving). Raised `strength` to the maximum (`1`) for every full-room-rebuild flow, which is a real, permanent improvement (fal.ai never had this parameter set at all before this bug report — every generation across the app was defaulting to `strength: 0.1`). User decided not to keep chasing 100% reliability on this one extreme source photo (a full floor-to-ceiling glass/metal shower enclosure) — accepted as a known hard edge case rather than actively worked further.

## Auth & security
- [ ] Replace fake client-side auth in `lamsa-auth.html` (login/register just write `localStorage.lamsa_user`, no server call) with a real backend: password hashing, session tokens, and server-side validation. **Now also a payments issue, not just a UX one**: credits are keyed by email with no verification that the visitor owns it — anyone can type any email into the credits modal and see/spend that email's balance. Fixing this properly means real accounts + server-verified sessions tied to the same email credits are keyed by.
- [ ] Add an auth guard check on `rearrange.html`, not just the main app — right now anyone can open it directly without logging in.
